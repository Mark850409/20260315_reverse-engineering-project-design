# 逆向工程推導專案設計 Skill

## 核心理念

從「已存在的事實（程式碼、DB、API）」推導「當初的設計決策」，重建缺失的完整文件套件。
分析重點不只是「這個系統怎麼設計的」，更延伸到「我們未來應該怎麼做」。

> 標記系統（全文一致使用）：
> - `[事實]` — 有程式碼/文件直接佐證
> - `[推測]` — 合理推論，無直接證據
> - `[待確認]` — 需開發者確認

---

## 第一步：輸入來源識別

| 輸入類型 | 分析重點 | 可信度 |
|---------|---------|--------|
| 程式碼 (Source Code) | 模組結構、命名慣例、耦合程度、設計模式 | ⭐⭐⭐⭐⭐ |
| API 文件 / Swagger | 資源模型、操作語義、認證機制 | ⭐⭐⭐⭐⭐ |
| 資料庫 Schema / ORM | 資料模型、關聯、業務規則 | ⭐⭐⭐⭐⭐ |
| 已部署產品/網站 | 功能邊界、使用者流程、業務邏輯 | ⭐⭐⭐⭐ |
| 混合輸入 | 交叉驗證，可信度最高 | ⭐⭐⭐⭐⭐ |

**接收流程：**
1. 快速掃描輸入，確認輸入類型
2. 告知使用者「偵測到 X，將分析並產出 [文件清單]」
3. 若輸入不足，主動詢問是否有補充材料
4. 確認使用者希望產出全套文件，還是特定幾份

---

## 第二步：系統性分析框架

依序執行以下四個維度的分析，作為所有文件的共同知識基礎：

### A. 邊界分析 (Boundary Analysis)
- 系統「做什麼」vs「不做什麼」
- 外部依賴與整合點（第三方 API、訊息佇列、外部服務）
- 使用者角色識別（admin、一般用戶、匿名訪客等）

### B. 結構分析 (Structure Analysis)
- 模組 / 服務 / Package 拆分邏輯
- 分層架構（展示層 / 業務層 / 資料層）
- 核心領域物件識別

### C. 行為分析 (Behavior Analysis)
- 主要業務流程（Happy Path）
- 例外處理模式與錯誤碼設計
- 狀態機識別（若有）

### D. 決策分析 (Decision Analysis)
- 技術選型推測（為何用這個框架 / 資料庫）
- 設計模式識別及其動機
- 可能的技術債與設計妥協點

### 常見模式速查

**架構模式識別：**
- MVC/MVP/MVVM：從命名規則（Controller/View/Model）識別
- 微服務：從 Docker Compose、API Gateway 設定識別
- 事件驅動：從 Message Queue、Event Handler 命名識別
- DDD：從 Domain、Repository、Aggregate 等命名識別

**資料庫 Schema 隱含業務邏輯：**
- deleted_at 欄位 → 軟刪除，有資料保留需求
- tenant_id / org_id → 多租戶 SaaS 架構
- created_by / updated_by → 稽核合規需求
- status enum → 核心業務狀態機

## 第三步：產出文件套件（預設產出全部）

分析完成後，依序產出以下 8 份文件。若使用者只需要部分文件，按需選擇。
**所有文件統一存放在專案根目錄的 docs/ 資料夾**，或依使用者指定路徑。

---

### 文件 1：需求規格書 PRD
**檔案名稱：** docs/PRD.md
> 閱讀 references/prd-template.md 取得完整模板與章節說明

**核心章節：**
1. 產品概述 — 從系統行為推導產品定位、目標市場
2. 使用者角色（Personas）— 每個角色的目標、痛點、使用頻率
3. 功能需求清單 — 從程式碼 / API 逆向整理，標記 Must Have / Should Have / Nice to Have
4. 非功能需求 — 效能、安全、可用性、可維護性（從架構選擇推測）
5. 已知限制與假設 — 技術限制、外部依賴限制
6. 待確認問題清單 — 需開發者確認的設計決策

---

### 文件 2：軟體設計文件 SDD
**檔案名稱：** docs/SDD.md
> 閱讀 references/sdd-template.md 取得完整模板與章節說明

**核心章節：**
1. 系統概覽 — 系統邊界、核心職責、版本資訊
2. 架構設計 — 分層架構說明、模組責任劃分、架構圖（Mermaid C4 Model，即文件 5）
3. 元件設計 — 每個主要模組的職責、介面、依賴
4. 資料設計 — 核心資料流向、轉換邏輯、快取策略
5. 安全設計 — 認證授權機制、敏感資料處理、攻擊面分析
6. 錯誤處理策略 — 錯誤碼設計、重試邏輯、降級策略
7. 效能考量 — 瓶頸識別、優化策略
8. 部署架構 — 容器化、環境配置、服務依賴

---

### 文件 3：API 規格文件
**檔案名稱：** docs/API-Spec.md
> 閱讀 references/api-spec-template.md 取得完整模板

**核心章節：**
1. API 總覽 — Base URL、版本策略、認證方式、Rate Limiting
2. 認證說明 — Token 取得方式、Header 格式、權限層級對照表
3. 通用規範 — 請求/回應格式、日期格式、分頁規範、排序規範
4. 錯誤碼定義 — HTTP 狀態碼對應、自訂錯誤碼、錯誤回應格式範例
5. Endpoint 目錄 — 按資源分組，每個 Endpoint 包含：
   - Method + Path
   - 功能說明
   - 權限要求（需要哪個 role / permission）
   - Request（Headers / Query Params / Body Schema）
   - Response（成功範例 + 失敗範例）
   - 注意事項 / 副作用

---

### 文件 4：Table Schema 文件
**檔案名稱：** docs/Schema.md
> 閱讀 references/schema-template.md 取得完整模板

**核心章節：**
1. 資料庫概覽 — 資料庫類型、版本、連線設定規範
2. 命名規範 — 資料表/欄位/索引命名規則
3. 資料表目錄 — 所有資料表一覽（含用途說明）
4. 資料表詳細定義 — 每張資料表包含：
   - 用途說明
   - 欄位清單（名稱、型別、必填、預設值、業務說明）
   - 主鍵 / 外鍵定義
   - 索引清單（名稱、欄位、類型、用途）
   - 約束條件（UNIQUE、CHECK 等）
   - 範例資料（2-3 筆）
5. ERD 實體關係圖（即文件 6）— Mermaid erDiagram 語法
6. 已知技術債 — Schema 設計上的改進建議

---

### 文件 5：系統架構圖
**嵌入在 SDD.md 的「架構設計」章節**，使用 Mermaid 按 C4 Model 三個層次繪製：

Context 圖（系統邊界）、Container 圖（主要容器/服務）、Component 圖（關鍵模組內部）。

每個層次的圖應清楚標示：節點名稱、技術棧、服務間資料流方向。
> 閱讀 references/erd-guide.md 第「架構圖範例」章節取得 Mermaid 語法範例

---

### 文件 6：ERD 實體關係圖
**嵌入在 Schema.md 末尾**，使用 Mermaid erDiagram 語法

完整呈現所有資料表及其關聯，包含：基數（一對一 / 一對多 / 多對多）、關聯說明標籤。
> 閱讀 references/erd-guide.md 取得進階繪製技巧與完整範例

---

### 文件 7：技術決策記錄 ADR
**檔案名稱：** docs/ADR.md
> 閱讀 references/adr-template.md 取得完整模板

每個技術決策記錄：狀態（事實/推測）、背景、考量選項（至少 2 個替代方案）、採用決策與原因、後果（正面+負面+需監控風險）。

---

### 文件 8：開發規範文件
**檔案名稱：** docs/Dev-Standards.md
> 閱讀 references/dev-standards-template.md 取得完整模板

每條規範使用三段式：
- 現況：「這個系統目前怎麼做」
- 標準：「未來新功能應遵循的規範」
- 原因：「為何這樣規定」

涵蓋：命名規範、架構規範、API 設計規範、資料庫規範、安全規範、測試規範。

---

## 第四步：品質檢查清單

- [ ] 推測內容標記 [推測]，事實標記 [事實]，不確定標記 [待確認]
- [ ] 文件間術語一致（同一概念不同文件用相同名稱）
- [ ] 每份文件開頭有執行摘要
- [ ] 架構圖、ERD 使用 Mermaid 可直接渲染
- [ ] API-Spec 每個 Endpoint 有完整的 Request / Response 範例（含成功與失敗）
- [ ] Schema 文件每張資料表有欄位的業務說明，不只是名稱
- [ ] SDD 安全章節涵蓋認證 + 授權 + 敏感資料處理
- [ ] ADR 每個決策說明了「為何不選其他方案」

---

## 第五步：互動式精煉

產出後主動詢問開發者確認設計意圖、補充資訊、確認技術債清單，格式：

「我已完成初步分析並產出文件，有幾點希望確認：[推測 X] / [功能 Z] / [模組 A 資訊不足] / [技術債 1~N]」
